# 온디바이스 AI 에이전트 시스템: 시니어 PM 종합 검토 보고서 (CPU-Only Plan Review)

## 1. 개요
본 문서는 시니어 프로젝트 매니저(PM) 관점에서, 사용자(NVIDIA GPU 부재 인텔 노트북) 환경에 맞춰 전면 개정된 CPU-Only(7B~8B 타겟) 기획 파일들(`블루프린트`, `인프라 아키텍처`, `워크플로우`, `PM 플랜`, `사전 요구사항` 등)의 **타당성, 완전성, 하드웨어 리스크**를 총체적으로 검토하고, 안정적인 프로젝트 완수를 위한 최종 보완 및 조율 방향을 제안합니다.

## 2. 개정된 기획 문서 종합 평가 (Evaluation)

### 2.1 타당성 (Feasibility) 평가: ⭐⭐⭐⭐ (우수함)
거대 20B 모델과 무거운 가상화(Docker)를 걷어내고, Ollama(AVX 가속)와 Qwen2.5-7B Coder, In-process LanceDB를 택한 점은 **현재 하드웨어 제약 하에서 온디바이스 AI를 동작시킬 수 있는 유일무이하고 가장 타당한 아키텍처**입니다. 
System RAM 여유분(최소 16GB 필요)을 정확히 인지하여 OOM을 메모리 스왑과 CPU 스로틀링 리스크로 조기 변환한 대처가 훌륭합니다.

### 2.2 일정 및 순서(Phase) 설계 평가: ⭐⭐⭐⭐ (양호하나 병목 주의)
Phase 1(코어 엔진)부터 Phase 4(Next.js 관제 UI)까지의 순서도는 자연스러우나, **CPU 환경이라는 특수성**을 간과하면 Phase 2(LightRAG 데이터 주입)와 Phase 3(Code Agent 실행)를 동시에 진행할 경우 노트북이 멈추는 프리징(Freezing) 현상이 발생할 수 있습니다.

## 3. 시니어 PM 핵심 검토 및 제안 (Risk & Alternatives)

문서 내용은 훌륭하나, 실제 프로젝트 착수 시 **'실패확률' 범위를 방어하기 위해 다음 3가지 핵심 제안(수정안)을 강제**합니다. 모든 결정사항은 프로젝트 생존율을 높이기 위해 비용/보안을 고려하였습니다.

### 🎯 제안 1: RAG 문서 Ingestion의 "비동기 배치 작업화" (워크플로우 수정 권고)
*   **리스크 (심각도-매우 높음)**: 사용자가 AI와 대화 중에 백그라운드에서 LightRAG가 7B 모델을 돌려 문서의 지식 그래프(Knowledge Graph)를 생성하려고 시도한다면, CPU 스레드 충돌로 LLM 응답이 생성되지 않고 타임아웃 뻗어버릴 확률 100%입니다.
*   **대체제 및 해결 (Mitigation)**: 
    *   FastAPI의 백그라운드 태스크나 `Celery` 같은 워커 큐를 분리하세요. 
    *   **사용자 대화(Chat) 중에는 문서 굽기(Ingestion)를 강제로 일시 정지(Pause)**하는 `스레드 락(Thread Lock)` 플래그를 아키텍처에 추가해야 코스트를 통제할 수 있습니다.

### 🎯 제안 2: Deno 샌드박스의 언어적 불일치 해소 (인프라 수정 권고)
*   **리스크 (심각도-중간)**: Code-Agent로 추천된 `Qwen2.5-7B-Coder`는 십중팔구 **Python 코드**를 가장 능숙하게 짜냅니다. 그러나 현재 설계된 보안 샌드박스는 V8 엔진 기반의 `Deno` (JS/TS 전용)입니다. Agent가 짠 파이썬 코드를 Deno로 우회해서 돌리려면 파서(Parser)를 작성해야 하는 추가 비용(Cost)과 복잡도 리스크가 심합니다.
*   **대체제 (Alternative)**: 
    *   보안적 리스크를 피하면서 파이썬 코드를 로컬에서 격리하는 가장 최신의 대체제인 **`uv run --isolated`** 명령어 스크립트 기반 격리(Chroot jail) 생태계를 구축하거나, 백엔드 내에서 `WebAssembly(Pyodide)`를 임베딩하여 파이썬을 가상 메모리 내에서 돌리는 구조로 블루프린트를 수정할 것을 제안합니다.

### 🎯 제안 3: 'Phase 1' 마일스톤에 철저한 "속도(TPS) 벤치마크" 장벽 추가
*   **리스크 (심각도-높음)**: 7B 모델 설치가 끝났다고 바로 RAG (Phase 2) 로 넘어가선 안 됩니다. 노트북 발열 상태에서 토큰 처리 속도가 나오지 않는다면 프로젝트는 그 용도를 잃습니다.
*   **추가 설계 진행 (Cost = 0)**: 
    *   Phase 1 완료 시점에 반드시 **Token Per Second(TPS)**를 측정하는 스킵 불가 벤치마크를 둡니다.
    *   *합격 기준*: 최소 **5~7 TPS** 미달 시, Qwen2.5-7B 에서 `Qwen2.5-3B` 체급으로 모델을 즉시 다운그레이드하는 오토 스케일다운 룰을 Phase 0 단계 체크리스트에 엄격히 구속시킵니다.

## 4. 최종 결론 및 넥스트 액션 안내
현 문서의 구상과 설계는 **NVIDIA GPU 없는 랩탑에서도 보안 및 AI 에이전트의 구동 한계를 시험해 볼 수 있는 한계 돌파형(Edge-case) 아키텍처로써 합격점**입니다. 

상기 제안된 3가지 방어 로직(비동기 락, 파이썬 전용 샌드박스 uv 도입, TPS 마일스톤)을 기존 문서들에 업데이트하는 작업을 승인하신다면, 제가 문서를 최종적으로 가다듬겠습니다.
아니면, 이 상태의 설계로 타당성을 확정 짓고 **가장 시급하게 CPU의 숨통이 트이는지 확인하기 위한 Phase 1 (Ollama 모델 로드 및 FastAPI + uv 뼈대 결합) 실무 코딩**에 돌입하셔도 좋습니다.

어느 쪽으로 진행할지 지시하여 주십시오.
