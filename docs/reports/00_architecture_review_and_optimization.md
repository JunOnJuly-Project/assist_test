# 온디바이스 AI 에이전트 시스템: 시니어 아키텍처 설계 및 검토 보고서 (Review & Optimization)

## 1. 개요 및 검토 목적
본 문서는 작성된 블루프린트, 인프라 아키텍처, 워크플로우 등 기존 문서들을 종합적으로 검토하고, **시니어 아키텍처 설계자** 관점에서 1) 타당성 2) 최적화 상태를 평가하며, 3) 수정할 부분 4) 추가할 부분을 구체적인 레퍼런스와 함께 피드백 및 설계로 제시합니다.

## 2. 평가 및 분석 내역

### 2.1 타당성 검토 (Feasibility)
*   ** LLM 엔진 및 16GB VRAM 제약**: 20B 파라미터 모델을 4-bit 양자화 시 최소 12~13GB VRAM을 요구합니다. 기존 문서에서 제시한 `vLLM`은 PagedAttention의 오버헤드와 Window 로컬 환경에서의 포팅 한계로 16GB VRAM 장비에서 **Out-of-Memory(OOM)** 위험이 매우 큽니다.
    *   *평가*: 조건부 타당. Windows 환경을 고려했을 때 vLLM보다는 메모리 CPU-GPU 오프로딩(Off-loading)을 지원하는 Native C/C++ 기반의 `llama.cpp` 혹은 `Ollama` 서빙으로 전면 교체하는 것이 타당합니다.
*   **데이터 파이프라인 (완전 오프라인 프라이버시 충돌)**: 기존 언급된 `LlamaParse`는 뛰어난 성능을 보이나 API 키를 요구하는 클라우드 SaaS일 확률이 높습니다. '완전한 온디바이스(Privacy)' 비전에 위배될 소지가 있습니다.

### 2.2 최적화 상태 (Optimization)
*   **Vector DB 및 도커 오버헤드**: 로컬 환경에서 `Qdrant`와 Code Sandbox를 위해 다수의 Docker 컨테이너(Docker Desktop)를 띄울 경우, Windows 커널(WSL2) 및 메모리 예약으로 인하여 정작 LLM 추론에 사용되어야 할 System RAM과 리소스가 크게 낭비됩니다.
    *   *최적화 상태*: 보통. 무거운 도커 기반이 아닌, 임베디드형(In-memory, Serverless) DB 컴포넌트 적용 등 경량화 최적화가 필수적입니다.

### 2.3 수정할 부분 (Modifications)
1.  **엔진 일원화**: LLM 서빙 엔진을 `Ollama` (내부적으로 llama.cpp 기반)로 일원화합니다.
    *   *리스크*: Ollama 생태계 제약.
    *   *대체제*: ExLlamav2.
    *   *코스트*: 오픈소스 무료, 로컬 OS 이식성 극대화 최저 비용.
2.  **경량화 벡터 DB로 교체**: Qdrant 대신 프로세스 내장형(embedded)인 **`LanceDB`** 또는 순수 SQLite 구조의 멀티모달 DB로 대체합니다.
    *   *리스크*: 엔터프라이즈급 대용량(수백만 건) 확장에서의 속도 저하. 하지만 개인(온디바이스) 규모에서는 타격 없음.
    *   *코스트/보안*: 완전 오프라인 파일시스템 I/O. Docker 리소스 낭비 없음 (RAM -2GB 이상 절약 효과).
3.  **오프라인 전용 고성능 문서 파서**: `LlamaParse`를 폐기하고, 피디에프의 테이블과 구조를 AI로 인식하는 로컬 오픈소스 **`Docling`** 혹은 **`marker-pdf`**로 전면 수정합니다.
4.  **Code Sandbox**: 무거운 백그라운드 Docker 엔진 대신, WebAssembly(WASM) 기반 컴파일된 런타임인 **`Pyodide`**를 백엔드 노드나 클라이언트에서 띄워 코드를 평가하게 함으로써, 해킹(OS 탈출) 방지와 제로보팅 메모리 오버헤드를 동시에 달성합니다.

### 2.4 추가할 부분 (Additions)
1.  **시맨틱 캐싱 (Semantic Caching)** 추가
    *   *내용*: `GPTCache` 기법 도입. 사용자가 동일하거나 의미상 매우 유사한 질문을 연속해서 던질 경우, 20B 모델 추론 파이프라인(가장 비싼 연산)을 타지 않고 즉시 DB 캐시에서 꺼내 응답속도를 10배 이상 높임.
    *   *코스트 및 리스크*: 캐시 DB 생성 시 약간의 스토리지 점유율 비용이 들지만 리소스 절감 압도적.
2.  **RAG용 LLM 교정 (Query Rewriting)** 추가
    *   *내용*: 문서 검색 성능을 최대치로 끌어올리기 위해 사용자의 짧은 질문을 검색에 최적화된 긴 키워드로 재작성(Expansion)하는 파이프라인.

## 3. 종합 결론
현재의 아키텍처는 "기능의 나열"로는 최상위급 수준이나, **"제한된 VRAM 16GB 이하의 온디바이스/Windows 환경"** 이라는 실물 하드웨어적 관점에서 볼 때 리소스 병목(Bottleneck)이 발생할 확률이 100%입니다.
본 리뷰의 제안 사항(WASM 샌드박스, LanceDB 임베드형 DB, Ollama, 시맨틱 캐싱)을 통해 동일한 기능 스펙을 제공하면서도 **리소스 사용량(RAM/VRAM)을 최대 40% 이상 경감**하는 방향으로 하위 문서들을 일괄 수정할 것입니다.
