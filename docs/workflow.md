# 온디바이스 AI 에이전트 시스템: 사용자 및 데이터 워크플로우 (Workflow) - [최종 합의: CPU-Only 통합본]

## 1. 개요
3인의 시니어 위원회 합의 사항(`Thread-Lock`, `uv isolated 파이썬 샌드박스`, `SmolAgents`)이 적용된 최종 에이전트 워크플로우입니다. 인텔 랩탑의 CPU 스로틀링 셧다운을 방어하는 "작업 교차 통제(Mutex)" 로직이 철저히 명세되었습니다.

## 2. 메인 에이전트 워크플로우 (우선순위: 대화)

### [Phase 0: 캐시 및 락 획득 인터셉터 계층]
*   **1단계. Thread-Lock 획득 검사**: 채팅 API 진입 즉시 시스템 글로벌 락(Lock)을 획득합니다. (만약 백그라운드 봇이 문서 요약 중이라면 즉각 일시정지(Pause) 신호를 보냄).
*   **2단계. 시맨틱 캐시(GPTCache) 및 보안(ProtectAI)**: 안전한 프롬프트인지 확인 후, 캐시에 동일 질문이 있으면 즉시 응답(CPU 사용 안 함).

### [Phase 1: 에이전트 브레인 및 LightRAG]
*   **3단계. Code-Agent 사고 및 액션 결정 (Ollama)**: 
    *   CPU의 물리 코어만 사용하여 7B 코딩 모델(Qwen2.5-Coder 등)이 현재 필요한 행동을 파이썬 코드 형태로 뱉어냄 (예: `search_rag("금융")`).
*   **4단계. LightRAG 조회 (하이브리드)**:
    *   내장된 LanceDB에서 관련 문구 조각(Vector)과 개체 연결망(Knowledge Graph)을 병합 조회하여 에이전트 메모리에 추가.

### [Phase 2: 격리된 파이썬 실행 샌드박스 액션]
*   **5단계. 샌드박스 연산 (`uv run --isolated`)**:
    *   에이전트가 만든 복잡한 계산식을 완전히 독립된 `uv` 가상환경(명령어 레벨)에 마운트하여 실행 (도커 불가, Deno 폐기). 
    *   호스트 파일 시스템 직접 공격 시도 원천 차단.
*   **6단계. 결과 반환 및 에러 자동 치유**: 
    *   실행 실패 시 에러가 STDOUT에 남으면 LLM이 2회까지 코드 자체 수정 시도. (3회 초과 시 CPU 보호를 위해 강제 취소 및 에러 반환 폴백).

### [Phase 3: 응답 렌더링 및 자원 해제]
*   **7단계. 스트리밍 및 락 해제 (Unlock)**: 
    *   Next.js 화면으로 UI 토큰 스트리밍. 발송 직후 글로벌 `Lock`을 해제하여, 멈춰있던 백그라운드 문서 Ingestion 작업이 다시 CPU를 쓸 수 있게 길을 터줌.

## 3. 지식 Ingestion 워크플로우 (단속적 백그라운드 수집)
사용자에게 방해되지 않도록 설계된 '눈치 보는' 인제스천 과정입니다.

1.  **변동 감지 및 파싱**: 로컬 파일 변경 시 `Docling`이 마크다운으로 추출.
2.  **Lock 확인 루프**: 추출을 위해 7B 모델에 엔티티 뽑기 명령을 내리기 전 `Lock` 상태 확인. 사용자가 타이핑 중이거나 대화 중이면 절대 넘기지 않음.
3.  **점진적(Lazy) 청킹 및 인덱싱**: 짧은 주기로 문서를 분할하여 임베딩 및 LanceDB에 적재하며, 노트북 팬이 미친 듯이 돌면(온도 상승 조건 등) 10초간 `sleep()` 휴식.
